## 环境准备

### 必需环境

| 依赖 | 用途 | 获取方式 |
|------|------|----------|
| Python 3.x | Level 2 解包、卡刷包处理等 | [python.org](https://www.python.org/downloads/) |
| Java（JRE/JDK） | 签名 FOTA 卡刷包（pack_level1_fota.bat） | [java.com](https://www.java.com) |
| 管理员权限 | 文件系统操作 | 右键脚本 → 以管理员身份运行 |

### 安装 Python 注意事项

安装时必须勾选 **"Add Python to PATH"**，否则脚本无法找到 Python。

```
☑ Add Python 3.x to PATH   ← 必须勾选此项
```

安装完成后验证：
```cmd
python --version
```

### 运行脚本的正确方式

所有 `.bat` 脚本均需要管理员权限。**推荐方式：右键 → 以管理员身份运行**，避免 UAC 弹出新窗口导致环境变量丢失。

> ⚠️ 直接双击运行时，脚本会通过 VBScript 重新启动一个系统级 cmd，该进程无法读取用户级 PATH，导致 Python 找不到。

## 判断固件类型

在解包之前或解包后，建议先判断固件的分区结构类型，以便选择正确的操作路径。

### 是否有 super 分区

super 分区是 Android 10 引入的动态分区容器，内部包含 system、vendor、product 等子分区。判断方法如下：

**方法一：查看 Level 1 解包结果（最直接）**

完成 Level 1 解包后，查看 `level1\` 目录：

```
有 super.PARTITION               → 使用动态分区（Android 10+）
                                   Level 2 必须选 python 方式（方式2）

无 super.PARTITION，但有：
  system.PARTITION
  vendor.PARTITION               → 使用传统独立分区（Android 9 及以下）
  product.PARTITION                Level 2 可选 vortex 或 python 方式
```

**方法二：查看 image.cfg**

用记事本打开 `level1\image.cfg`，在 `[LIST_VERIFY]` 段中查找：

```ini
[LIST_VERIFY]
file="super.PARTITION"    main_type="PARTITION"    sub_type="super"
↑ 有此行 = 动态分区

file="system.PARTITION"   main_type="PARTITION"    sub_type="system"
↑ 有此行（无 super）= 传统分区
```

**方法三：Android 版本参考（辅助判断，不绝对）**

| Android 版本 | super 分区 | Level 2 解包方式 |
|---|---|---|
| Android 9 及以下 | ❌ 无 | vortex 或 python 均可 |
| Android 10 | ✅ 通常有 | 必须选 python（方式 2） |
| Android 11 及以上 | ✅ 基本都有 | 必须选 python（方式 2） |

> ⚠️ 部分厂商在 Android 10 上仍使用传统分区，以实际文件为准，版本仅供参考。

**方法四：WSL / Linux 下验证（深度确认）**

```bash
# 查看文件格式
file level1/super.PARTITION
# 输出含 "Android sparse image" 说明是 sparse 格式的 super

# 查看 super 内部分区信息
bin/windows/super/lpunpack --show-info level1/super.PARTITION
# 能正常输出 system_a、vendor_a 等分区信息即确认为合法 LP 格式
```

---

### 分区镜像格式识别

Level 1 解包出的 `.PARTITION` 文件可能有以下几种格式，会影响 Level 2 的处理方式：

| 格式 | 文件头（十六进制） | 说明 | 处理方式 |
|------|--------------------|------|----------|
| Android Sparse | `3A FF 26 ED` | 压缩的 ext4，节省空间 | simg2img 转换后再解包 |
| Raw ext4 | `53 EF`（偏移 0x438）| 标准 ext4 文件系统 | 直接解包 |
| LP（super）| `67 44 6C 61` | 动态分区容器 | lpunpack 拆分后再解包 |
| Android Boot | `41 4E 44 52 4F 49 44 21` (`ANDROID!`) | boot/recovery 镜像 | aik 解包 |
| DTB | `D0 0D FE ED` | 设备树二进制 | dtc 反编译 |
| Amlogic 多 DTB | 自定义头 | 多个 DTB 打包在一起 | dtbSplit 拆分后 dtc 反编译 |

在 Windows 下可用 **HxD** 或 **010 Editor** 打开文件查看文件头来确认格式。

## Amlogic 线刷包处理

### unpack.bat — 解包

解包流程分三个 Level，依次执行。

---

#### Level 1 — 解包线刷 img

**前提：** 将 `.img` 线刷包放入 `in\` 目录。

**操作：** 运行 `unpack.bat` → 输入 `1` → 输入文件名（不含 `.img` 后缀）

**原理：**
```
bin\windows\AmlImagePack -d "in\文件名.img" level1\
```

**输出：** `level1\` 目录下所有分区文件

```
level1\
├── projectname.txt          ← 记录文件名，供后续脚本使用
├── image.cfg                ← 打包配置文件（LIST_NORMAL/LIST_VERIFY）
├── bootloader.PARTITION     ← u-boot 引导
├── _aml_dtb.PARTITION       ← 设备树（DTB）
├── boot.PARTITION           ← Android boot 分区
├── recovery.PARTITION       ← recovery 分区
├── system.PARTITION         ← system 分区（可能是 sparse ext4）
├── vendor.PARTITION         ← vendor 分区
├── product.PARTITION        ← product 分区（Android 10+）
├── super.PARTITION          ← 动态分区容器（Android 10+，含 system/vendor 等）
├── logo.PARTITION           ← 开机画面
├── DDR.USB                  ← USB 烧录必需（初始化内存）
├── UBOOT.USB                ← USB 烧录必需（u-boot）
└── ...
```

---

#### Level 2 — 解包 ext4 分区内容

**前提：** 已完成 Level 1。执行前建议先[确认是否有 super 分区](#是否有-super-分区)，以选择正确的解包方式。

**操作：** 运行 `unpack.bat` → 输入 `2` → 选择解包方式

```
1 - vortex    使用 imgextractor.exe，速度快，不依赖 Python
              ✅ 适用于传统分区（无 super）
              ❌ 不支持 super 动态分区

2 - python    使用 imgextractor.py，支持 sparse 自动转换
              ✅ 支持传统分区和 super 动态分区
              ⚠️  需要已安装 Python 且在 PATH 中
```

> 💡 如果不确定，统一选 2（python）更安全，兼容性更好。

**处理的分区：** `system` `vendor` `product` `odm` `oem` `system_ext` `odm_ext_a/b`

**sparse 格式自动处理（python 方式）：**
```
检测到 sparse 格式 → simg2img 转 raw → imgextractor.py 解包
普通 raw 格式     → imgextractor.py 直接解包
```

**super 动态分区处理（仅 python 方式）：**
```
simg2img 转 raw → lpunpack 拆分 → 解包各子分区
→ 写入 super_type.txt（2=单槽，3=双槽 A/B）
```

**输出：**
```
level2\
├── system\              ← /system 文件树，可直接修改
├── vendor\              ← /vendor 文件树
├── product\             ← /product 文件树
├── odm\                 ← /odm 文件树
└── config\
    ├── system_size.txt           ← 原始分区大小（打包时使用）
    ├── system_file_contexts      ← SELinux 标签（打包时使用）
    ├── system_fs_config          ← 文件权限/uid/gid（打包时使用）
    ├── super_size.txt            ← super 原始大小（如有）
    └── super_type.txt            ← super 类型（2 或 3）
```

---

#### Level 3 — 解包 boot / recovery / DTB / logo

**前提：** 已完成 Level 1。

**操作：** 运行 `unpack.bat` → 输入 `3`

**各组件处理流程：**

**boot / recovery：**
```
复制 .PARTITION 到 aik 目录 → unpackimg.bat 解包
→ ramdisk\ 移至 level3\boot\ramdisk\
→ split_img\ 移至 level3\boot\split_img\
```

**DTB（智能 fallback 查找顺序）：**
```
1. level1\_aml_dtb.PARTITION        优先使用独立 DTB 分区
2. level3\boot\split_img\boot.img-dtb
3. level3\boot_a\split_img\boot_a.img-dtb
4. boot.img-second（旧设备）
5. recovery 中的 dtb / second
```
找到后统一反编译为 `.dts` 文本：
```
dtc -I dtb -O dts → level3\devtree\single.dts   （单 DTB）
dtbSplit 拆分     → level3\devtree\dtb_0.dts 等  （多 DTB 容器）
```

**logo：**
```
imgpack -d → level3\logo\ 下多张 .bmp 图片
```

**输出：**
```
level3\
├── boot\
│   ├── ramdisk\                 ← ramdisk 文件树（可修改 init 脚本等）
│   └── split_img\
│       ├── boot.img-kernel      ← 内核
│       ├── boot.img-cmdline     ← 启动参数
│       ├── boot.img-base        ← 基地址
│       └── boot.img-pagesize    ← 页大小
├── recovery\
│   └── ramdisk\                 ← recovery 文件系统
├── devtree\
│   ├── single.dts               ← 整体 DTB 反编译结果
│   ├── dtb_0.dts                ← 拆分后各硬件配置
│   └── dtb_1.dts ...
└── logo\
    ├── bootup.bmp               ← 开机 logo
    ├── upgrade_bar.bmp          ← 升级进度条
    └── ...
```

---

### pack.bat — 打包

打包顺序：**Level 3 → Level 2 → Level 1**，最终在 `out\` 生成线刷包。

---

#### Level 3 — 重打 boot / recovery / DTB / logo

**操作：** 运行 `pack.bat` → 输入 `3`

**boot / recovery 打包：**
```
ramdisk\ + split_img\ 移回 aik → repackimg.bat → image-new.img
→ 再次 unpackimg 验证 → 移至 level1\boot.PARTITION
```

**DTB 打包（有大小判断）：**
```
多个 .dts → 各自编译为 .dtb → dtbTool 合并为多 DTB 容器
单个 .dts → dtc 直接编译

最终判断大小：
  > 192KB → gzip 压缩 → level1\_aml_dtb.PARTITION
  ≤ 192KB → 直接复制 → level1\_aml_dtb.PARTITION
```

**logo 打包：**
```
imgpack -r level3\logo → level1\logo.PARTITION
```

---

#### Level 2 — 重打 ext4 分区

**操作：** 运行 `pack.bat` → 输入 `2`

**普通分区（无 super）：**
```
make_ext4fs
  -s                              生成 sparse 格式
  -J                              启用 journal
  -L [分区名]                     卷标
  -T -1                           时间戳固定（保证可重现）
  -S config\system_file_contexts  SELinux 标签
  -C config\system_fs_config      文件权限/uid/gid
  -l [原始大小]                   严格按 config\system_size.txt
  -a [挂载点]
  level1\system.PARTITION         输出（覆盖原文件）
  level2\system\                  输入文件树
```

**super 动态分区（Android 10+）：**
```
各子分区 → make_ext4fs → .img
→ resize2fs -M 压缩到最小
→ lpmake 合并
  --metadata-slots 2（单槽）或 3（A/B 双槽，由 super_type.txt 决定）
  --virtual-ab --sparse
  → level1\super.PARTITION
```

> ⚠️ 打包前会检查总大小是否超出原始 super 容量，超出则报错退出，需要删减文件后重试。

---

#### Level 1 — 生成最终线刷包

**操作：** 运行 `pack.bat` → 输入 `1`

```
AmlImagePack -r level1\image.cfg level1\ "out\文件名.img"
```

**输出：** `out\文件名.img`，可直接用 **USB Burning Tool** 烧录。

## 其他 Amlogic 工具

### dump.bat — 从设备提取固件

通过 USB 连接处于 **mask 模式**（屏蔽模式/强制烧录模式）的 Amlogic 设备，直接读取各分区数据。

**前提：** 设备进入 mask 模式（通常短接主板上的 mask 点，或使用牙签捅 AV 口）

**操作：** 运行 `dump.bat`，按提示输入参数

**流程：**
```
1. 输入 bootloader 大小（多数设备为 4194304 字节）
2. 输入 DTB 大小（多数设备为 262144 字节）
3. 自动读取 bootloader 和 DTB
4. 反编译 DTB → 列出所有 .dts 文件
5. 选择包含分区表的 .dts 文件
6. 解析分区表 → 自动读取所有分区 → 保存到 dump\ 目录
```

**输出：** `dump\` 目录下各分区的 `.img` 文件

```
dump\
├── bootloader.img
├── dtb.img
├── boot.img
├── system.img
├── vendor.img
└── ...
```

---

### dump_to_aml.bat — dump 转线刷包

将 `dump.bat` 提取的分区镜像，重新打包为可用 USB Burning Tool 烧录的线刷包。

**前提：**
- 已完成 dump，`dump\` 目录下有分区镜像
- `level1\` 目录下需手动放置以下文件（来自同型号官方固件）：
  - `DDR.USB`
  - `UBOOT.USB`
  - `aml_sdc_burn.UBOOT`
  - `meson1.PARTITION`
  - `platform.conf`

**操作：** 右键管理员运行 `dump_to_aml.bat`

**流程：**
```
1. 复制 dump\ 下各分区镜像 → level1\*.PARTITION
2. 自动生成 image.cfg（LIST_NORMAL + LIST_VERIFY）
3. 解包 system/vendor 等 → level2\ （用于重建 ext4）
4. 重打 ext4 分区（修复文件系统）→ 更新 level1\
5. 解包 boot/recovery/DTB/logo → level3\
6. 重打 boot/DTB/logo → 更新 level1\
7. AmlImagePack 打包 → out\自定义名.img
```

> ⚠️ 暂不支持含 super 动态分区的设备（Android 10+ 动态分区）。

---

### pack_zip_to_aml.bat — 卡刷包转线刷包

将 Android 卡刷包（Recovery 刷机包，`.zip` 格式）转换为 Amlogic 线刷包。

**前提：**
- 将卡刷 `.zip` 放入 `in\` 目录
- `level1\` 下需手动放置：`DDR.USB`、`UBOOT.USB`、`aml_sdc_burn.UBOOT`、`meson1.PARTITION`、`platform.conf`
- 已安装 Python

**操作：** 运行 `pack_zip_to_aml.bat` → 输入文件名（不含 `.zip`）

**流程：**
```
1. 解压 zip 到 tmp\
2. 删除 META-INF、compatibility.zip 等无关内容
3. 处理 dat.br 格式的分区（brotli 解压 → sdat2img 还原 → img2simg 转 sparse）
4. 重命名各分区镜像为 .PARTITION 移至 level1\
5. 自动生成 image.cfg
6. AmlImagePack 打包 → out\文件名.img
```

**支持的输入格式：** 标准 AOSP OTA 包（含 `system.transfer.list` + `system.new.dat.br`）

---

### pack_level1_fota.bat — 生成 FOTA 卡刷包

将 Level 1 的分区文件重新打包为 FOTA OTA 卡刷包（`.zip` 格式，可在 recovery 中刷入）。

**前提：**
- 已完成 Level 1 解包
- 已安装 Python 和 Java
- 不支持含 super 动态分区的固件

**操作：** 运行 `pack_level1_fota.bat`，根据提示选择是否用 brotli 压缩

**流程：**
```
1. 复制基础文件框架（updater-script 模板等）到 tmp\
2. 处理各分区：
   - system/vendor 等 ext4 分区 → img2sdat 转换为 transfer.list + new.dat
   - 可选 brotli 压缩 → .new.dat.br
   - boot/recovery/logo/dtbo/vbmeta 直接复制
3. 自动生成 updater-script（包含各分区更新指令）
4. 7za 打包为 update_tmp.zip
5. zipsigner.jar 用 testkey 签名 → out\文件名_fota.zip
```

**输出：** `out\文件名_fota.zip`，可在兼容的 recovery 中刷入。

## Rockchip 固件处理

### rkunpack.bat — 解包

**操作：** 右键管理员运行 `rkunpack.bat`

#### Level 1 — 解包 Rockchip img

```
将 .img 放入 in\ 目录 → 选 1 → 输入文件名
rkImageMaker -unpack → firmware.img
afptool -unpack      → level1\Image\ 下各分区 .img
```

**输出：** `level1\Image\*.img`（boot.img、system.img、vendor.img 等）

#### Level 2 — 解包 ext4 分区（需要 Python）

处理逻辑与 Amlogic 相同，支持 super 动态分区，输出到 `level2\`。

#### Level 3 — 解包 boot / recovery

```
复制 level1\Image\boot.img → aik → unpackimg.bat
→ ramdisk\ 和 split_img\ 移至 level3\boot\
```

## Level 2 定制 — 文件系统内容

Level 2 对应设备运行时挂载的各个分区的文件系统，是最常用的定制层。

**重要规则：**
- **替换已有文件** → 直接覆盖，无需改权限文件
- **新增文件或目录** → 必须同步更新 `config\*_fs_config` 和 `config\*_file_contexts`
- **删除文件** → 直接删除，无需改权限文件

---

### 预装 / 删除应用

**应用目录说明：**

```
level2\system\app\              普通系统应用（READ_EXTERNAL_STORAGE 等普通权限）
level2\system\priv-app\         特权应用（可申请 INSTALL_PACKAGES 等 signature 级权限）
level2\vendor\app\              vendor 层应用（与硬件驱动相关）
level2\product\app\             product 层应用（部分机型有此分区）
level2\system\overlay\          运行时资源覆盖包（RRO，用于主题定制）
```

**删除预装应用：**

直接删除对应目录，无需修改其他文件：

```cmd
rmdir /s /q "level2\system\app\Browser2"
rmdir /s /q "level2\system\app\Email"
```

> 💡 删除应用可释放 super 分区空间，为新增内容腾出余量。

**替换已有应用（升级/降级）：**

直接用新版 APK 覆盖同名文件即可：

```
level2\system\app\Settings\Settings.apk  ← 直接覆盖替换
```

**新增应用（全新预装）：**

需要三个步骤，缺一不可：

第一步，创建目录并放入 APK：
```
level2\system\app\MyApp\MyApp.apk
```

第二步，在 `level2\config\system_fs_config` 末尾追加：
```
system/app/MyApp 0 0 755 capabilities=0x0
system/app/MyApp/MyApp.apk 0 0 644 capabilities=0x0
```

第三步，在 `level2\config\system_file_contexts` 末尾追加：
```
/system/app/MyApp(/.*)?    u:object_r:system_file:s0
```

**新增特权应用（priv-app）：**

特权应用还需要额外的权限白名单文件，否则 PackageManager 会拒绝授予特权权限：

第一步，在 `level2\system\etc\permissions\` 下创建 `privapp-permissions-MyApp.xml`：
```xml
<?xml version="1.0" encoding="utf-8"?>
<permissions>
    <privapp-permissions package="com.example.myapp">
        <permission name="android.permission.INSTALL_PACKAGES"/>
        <permission name="android.permission.DELETE_PACKAGES"/>
    </privapp-permissions>
</permissions>
```

第二步，在 `level2\config\system_fs_config` 追加：
```
system/priv-app/MyApp 0 0 755 capabilities=0x0
system/priv-app/MyApp/MyApp.apk 0 0 644 capabilities=0x0
system/etc/permissions/privapp-permissions-MyApp.xml 0 0 644 capabilities=0x0
```

第三步，在 `level2\config\system_file_contexts` 追加：
```
/system/priv-app/MyApp(/.*)?                           u:object_r:system_file:s0
/system/etc/permissions/privapp-permissions-MyApp\.xml u:object_r:system_file:s0
```

---

### 修改系统属性

系统属性有多个生效层次，优先级从高到低：

```
运行时设置（setprop）> build.prop > default.prop（ramdisk）
```

**build.prop 位置：** `level2\system\build.prop`

```properties
# ── 设备标识（影响 Play Store 兼容性检测）──
ro.product.model=自定义型号
ro.product.brand=自定义品牌
ro.product.manufacturer=自定义厂商
ro.product.device=设备代号
ro.build.fingerprint=品牌/设备/版本号...   # 修改此项可伪装设备

# ── 显示 ──
ro.sf.lcd_density=240              # DPI：160/213/240/320/480
ro.display.size=1920x1080          # 部分固件使用此项覆盖分辨率

# ── 语言与地区 ──
ro.product.locale=zh-CN
persist.sys.timezone=Asia/Shanghai
persist.sys.language=zh
persist.sys.country=CN

# ── ART 虚拟机性能 ──
dalvik.vm.heapsize=512m            # 单个应用最大堆内存
dalvik.vm.heapgrowthlimit=256m     # 普通应用堆增长上限
dalvik.vm.heapstartsize=16m        # 初始堆大小

# ── 网络 ──
net.dns1=8.8.8.8                   # 默认 DNS
net.dns2=8.8.4.4

# ── 调试 ──
ro.debuggable=1                    # 开启调试（生产固件为 0）
ro.secure=0                        # 关闭安全限制（允许 root adb）
persist.sys.usb.config=mtp,adb    # 默认 USB 模式

# ── 性能模式 ──
persist.sys.performance=4          # 部分 Amlogic 固件支持
ro.config.low_ram=false            # 禁用低内存模式

# ── 多媒体 ──
media.stagefright.enable-player=true
ro.media.dec.avi.enabled=1
```

**vendor 属性（硬件相关）：** `level2\vendor\build.prop`

```properties
ro.vendor.build.version.release=11
vendor.display.resolution=1080p
```

> ⚠️ `ro.build.fingerprint` 修改后可能影响 SafetyNet / Play Integrity 验证。

### 添加与替换系统文件

**系统字体：**

```
level2\system\fonts\               ← 替换或新增 .ttf / .otf 字体文件
level2\system\etc\fonts.xml        ← 修改字体映射配置
level2\system\etc\font_fallback.xml ← 备用字体配置
```

替换字体后无需修改权限文件（替换已有文件）。新增字体文件需在 `system_fs_config` 中追加：
```
system/fonts/MyFont.ttf 0 0 644 capabilities=0x0
```

**系统铃声 / 提示音 / 闹钟音：**

```
level2\system\media\audio\ringtones\    ← 铃声
level2\system\media\audio\notifications\ ← 通知音
level2\system\media\audio\alarms\       ← 闹钟
level2\system\media\audio\ui\           ← 界面音效
```

直接替换 `.mp3` / `.ogg` 文件，文件名保持一致则无需改权限文件。

**系统壁纸：**

```
level2\system\media\default_wallpaper.jpg    ← 默认壁纸（部分机型）
level2\product\media\default_wallpaper.jpg   ← Android 10+ 壁纸位置
```

**CA 证书（安装内置证书）：**

```
level2\system\etc\security\cacerts\   ← 存放 .0 格式的 PEM 证书
```

证书文件名为 `证书hash.0`，可用 OpenSSL 生成：
```bash
openssl x509 -inform PEM -subject_hash_old -in cert.pem | head -1
# 输出的 hash 值即为文件名前缀，如 a3418fda.0
```

**系统 so 库：**

```
level2\system\lib\        ← 32位库
level2\system\lib64\      ← 64位库
level2\vendor\lib\        ← vendor 32位库
level2\vendor\lib64\      ← vendor 64位库
```

直接替换同名 `.so` 文件，确保架构匹配（ARM/ARM64）。

```
127.0.0.1  tracker.example.com
```

### 修改分区挂载参数

**fstab 位置：** `level2\system\etc\fstab.amlogic`（文件名因设备不同可能有差异）

```
# 格式：设备路径  挂载点  文件系统  挂载选项  fs_mgr_flags
/dev/block/system   /system   ext4   ro,barrier=1   wait,avb=vbmeta

# 常见修改：将 system 改为读写挂载（去除 ro）
/dev/block/system   /system   ext4   rw,barrier=1   wait
```

> ⚠️ 修改挂载参数可能影响系统安全验证（AVB/dm-verity），如遇启动失败需确认 vbmeta 配置。


### 修改权限与 SELinux 标签

**权限声明文件：** `level2\system\etc\permissions\`

```
platform.xml              ← 系统权限组定义
privapp-permissions-*.xml ← 特权应用权限白名单
com.android.*.xml         ← 各系统功能的特性声明
```

**手动修改文件权限条目（fs_config 格式说明）：**

```
路径  uid  gid  权限八进制  capabilities=0x能力值

示例：
system/bin/su  0  0  6755  capabilities=0x0
               ↑  ↑  ↑↑↑↑
               |  |  ||||── 其他用户权限（5=读+执行）
               |  |  |||─── 同组用户权限（5=读+执行）
               |  |  ||──── 文件所有者权限（7=读写执行）
               |  |  |───── setuid/setgid 位（6=setuid+setgid）
               |  └──────── gid（0=root，1000=system，2000=shell）
               └─────────── uid（0=root，1000=system）
```

**常用 uid/gid 对照：**

| uid/gid | 含义 |
|---------|------|
| 0 | root |
| 1000 | system |
| 1001 | radio |
| 1002 | bluetooth |
| 2000 | shell |
| 2001 | cache |

## Level 3 定制 — boot / DTB / logo

Level 3 对应启动相关的底层组件，风险高于 Level 2，修改前务必备份。

---

### 修改开机 logo

`level3\logo\` 下包含多张 BMP 图片：

```
bootup.bmp           ← 主开机画面（上电后第一张）
upgrade_logo.bmp     ← 刷机过程中的 logo
upgrade_bar.bmp      ← 刷机进度条背景
upgrade_error.bmp    ← 刷机失败提示画面
upgrade_success.bmp  ← 刷机成功提示画面（部分固件）
```

**制作替换图片的要求：**

- 尺寸必须与原图**完全一致**（用图像查看器确认原图宽×高）
- 格式：**32位 BMP（RGBA 或 BGRA）**，部分旧设备为 24位 RGB
- 导出方法（Photoshop）：文件 → 存储为 → BMP → 选择 32位

> ⚠️ 尺寸哪怕只差 1 像素也会导致显示错位或黑屏卡住。

---

### 修改 DTB（设备树）

编辑 `level3\devtree\` 下的 `.dts` 文件。

**确认目标 DTB：**

多 DTB 固件中每个 `.dts` 文件开头都有设备标识：

```dts
/ {
    model = "Amlogic Meson S905X3 Board";  ← 确认是否为目标设备
    compatible = "amlogic,s905x3", "amlogic,meson";
```

**常见可修改项：**

```dts
/* CPU 频率（单位 Hz） */
clock-frequency = <1908000000>;   /* 1.908 GHz */

/* DDR 频率 */
max-freq = <912000000>;           /* 912 MHz */

/* GPU 频率 */
assigned-clock-rates = <666000000>;

/* HDMI 相关 */
hdmi_4k2k_support = <1>;         /* 开启 4K 支持 */
hdmi_vesa_only = <0>;

/* 串口波特率（调试用） */
current-speed = <115200>;

/* 温控阈值（单位：千分之一摄氏度） */
trip-point@0 {
    temperature = <70000>;        /* 70°C 开始降频 */
    hysteresis = <2000>;
    type = "passive";
};
trip-point@1 {
    temperature = <85000>;        /* 85°C 紧急降频 */
    type = "hot";
};

/* LED 配置 */
amlogic,led_gpio = <&gpio GPIODV_24 GPIO_ACTIVE_HIGH>;

/* 内存布局（不建议修改） */
linux,usable-memory = <0x0 0x100000 0x0 0x3f00000>;
```

**多 DTB 修改策略：**

```
只有 single.dts    → 直接修改 single.dts
有 dtb_0/1/2.dts  → 确认硬件对应哪个，只改对应文件
                    如不确定，全部改成相同的值
```

> ⚠️ 修改频率等参数前先记录原始值，一旦无法启动可以恢复。

---

### 修改 ramdisk（init 脚本）

`level3\boot\ramdisk\` 对应设备启动时的根文件系统，在 `system` 分区挂载之前就已加载。

**常见可修改文件：**

```
init.rc                 ← 系统主启动脚本
init.amlogic.rc         ← Amlogic 平台启动脚本
init.[设备].rc          ← 设备专属启动脚本
default.prop            ← 最早生效的系统属性
fstab.[平台]            ← 分区挂载配置
ueventd.rc              ← 设备节点权限配置
```

**default.prop 常用修改：**

```properties
# 开启调试模式与 ADB root
ro.debuggable=1
ro.secure=0
persist.sys.usb.config=mtp,adb

# 关闭 SELinux（临时调试用，不建议生产环境）
androidboot.selinux=permissive
```

**init.rc 语法示例（新增服务）：**

```rc
# 格式要求：缩进必须为 4 个空格，不能用 Tab
service myservice /system/bin/myservice
    class main
    user system
    group system
    oneshot

# 在启动时执行命令
on property:sys.boot_completed=1
    start myservice
    setprop my.custom.prop value
```

> ⚠️ init.rc 任何语法错误（多余空格、Tab 缩进、拼写错误）都会导致设备无法开机，修改前**必须备份**。

---

### 修改启动参数

**文件位置：** `level3\boot\split_img\boot.img-cmdline`

用文本编辑器打开，是一行内核参数：

```
console=ttyS0,115200 androidboot.selinux=enforcing quiet ...
```

**常见修改：**

```
# 开启串口调试输出（去掉 quiet，保留 console）
console=ttyS0,115200

# 关闭 SELinux（调试用）
androidboot.selinux=permissive

# 关闭 dm-verity 验证（允许修改 system 后仍能启动）
androidboot.veritymode=eio

# 传递自定义参数给 init
androidboot.myprop=myvalue
```

> ⚠️ 修改 `dm-verity` 相关参数可能导致安全验证失败，根据固件的 vbmeta 配置决定是否生效。

---

### 替换内核（高级）

**文件位置：** `level3\boot\split_img\boot.img-kernel`

直接替换该文件即可，打包时 aik 会将新内核重新打包进 boot.img。

**替换前必须确认：**

```
1. 新内核针对相同 SoC 型号编译（如同为 S905X3）
2. 内核版本与 Android 系统版本兼容
   Android 9  → Linux 4.9
   Android 11 → Linux 5.4
   Android 12 → Linux 5.10
3. 内核配置中的 DTB 与设备硬件匹配
4. 内核模块（.ko 文件）版本与内核一致
   → 模块通常在 level2\vendor\lib\modules\
```

> ⚠️ 内核替换风险极高，内核与模块版本不匹配会导致设备无法启动或功能异常，仅适合有嵌入式开发经验的用户。

---

### 解包方式对打包结果的影响

Level 2 解包提供 vortex（方式1）和 python（方式2）两种方式，对最终打包结果的影响如下：

| 对比项 | vortex 方式 | python 方式 | 影响 |
|--------|------------|------------|------|
| 文件树内容 | 一致 | 一致 | ✅ 无影响 |
| fs_config 根目录条目 | 无前导斜杠 | 有前导斜杠（`/lost+found`）| ✅ make_ext4fs 兼容，无影响 |
| file_contexts 去重 | 不去重 | 去重排序 | ✅ 重复条目标签相同时无影响 |
| super 分区支持 | ❌ 不支持 | ✅ 支持 | ⚠️ 有 super 时必须用 python |
| space.txt 生成 | 不生成 | 生成（路径含空格时）| ✅ 打包不使用此文件，无影响 |

**结论：** 两种方式打包的功能结果等价，不会导致设备无法开机。有 super 分区时必须选 python 方式。

### 操作后检查清单

打包前逐项确认：

```

□ 新增文件已在对应的 *_fs_config 中添加权限条目
□ 新增文件已在对应的 *_file_contexts 中添加 SELinux 标签

□ 未修改 level2\config\ 下的任何元数据文件
□ super 分区固件：确认修改后总体积未超出 super_size.txt 的限制

Level 3 相关：

□ init.rc 修改后使用 4 空格缩进（不含 Tab）
□ boot.img-cmdline 修改后仍为单行（无换行符）
□ 已备份修改前的原始 .dts 和 init*.rc 文件

通用：
□ 未修改 bootloader.PARTITION
□ 保留了原始固件（in\ 目录下的 .img 文件）备份
```

## 常见问题

**Q: Level 1 提示 "Can't find images"**  
A: 未在 `in\` 目录放置 `.img` 文件，或 `in\` 目录不存在。手动创建 `in\` 文件夹并将线刷包放入，输入文件名时不含 `.img` 后缀。

**Q: Level 2 提示 "Python not installed"**  
A: Python 未正确加入 PATH，或脚本通过 UAC 提权后丢失用户级 PATH。解决方法：**右键以管理员身份运行**（不要双击），或将 Python 路径加入系统变量（控制面板 → 系统 → 高级系统设置 → 环境变量 → 系统变量中的 Path）。

**Q: Level 2 vortex 方式解包失败，但 python 方式成功**  
A: 固件使用了 super 动态分区，vortex 方式（imgextractor.exe）不支持，切换为 python 方式（选 2）即可。

**Q: Level 2 python 方式提示 sparse 检测相关错误**  
A: `bin\windows\file` 工具判断格式失败，可手动用 HxD 检查文件头：`3A FF 26 ED` 开头为 sparse 格式，可手动运行 `bin\windows\simg2img.exe` 转换后再解包。

**Q: pack Level 2 提示 "Unable to create super image, recreated images are too big"**  
A: 定制后文件总大小超出原始 super 分区容量上限（保存在 `level2\config\super_size.txt`）。需要删减添加的内容，或删除一些原有的预装应用，减小各子分区体积后重试。

**Q: DTB 反编译有报错或警告**  
A: Amlogic 多 DTB 容器格式特殊，`dtc` 直接解析会有警告，属正常现象。只要 `dtbSplit` 拆分后单独编译没有报 ERROR 级错误即可继续，警告（WARNING）可忽略。

**Q: 打包后用 USB Burning Tool 烧录成功，但设备无法开机**  
A: 常见原因有三个：① `make_ext4fs` 在 Windows 下打包时文件权限丢失，建议切换 WSL/Linux 重新执行 Level 2 打包；② 新增文件未正确配置权限，系统启动时 SELinux 阻止访问；③ `init.rc` 或 `default.prop` 有语法错误。可尝试先恢复原版固件确认设备正常，再逐步排查。

**Q: 打包后 Level 1 提示 "Can't find image.cfg"**  
A: `level1\image.cfg` 丢失或被误删。该文件在 Level 1 解包时自动生成，重新执行 Level 1 解包即可恢复，不要手动删改该文件。

**Q: dump_to_aml.bat 缺少 DDR.USB 等文件**  
A: 这些文件是 u-boot 的 USB 烧录必需组件，必须来自同型号同芯片的官方固件，不可跨型号混用。请从对应型号的官方线刷包中提取后复制到 `level1\` 目录。

**Q: pack_level1_fota.bat 打包完成但刷机时报错**  
A: FOTA 包使用 testkey 签名，部分 recovery 只接受官方 key 签名。此外，updater-script 中的分区名称（如 `/dev/block/system`）需与目标设备实际的块设备路径一致，否则刷机脚本找不到目标分区。

**Q: Rockchip 打包时提示 chip 无效**  
A: 芯片代码必须以 `RK` 开头，参考 rkpack.bat 内的对照表输入，如 `RK3568`、`RK312X`，注意大小写。

## 注意事项

### 操作安全

1. **操作前务必备份原始固件**，将 `in\` 目录下的原始 `.img` 另存一份，确保出现问题时可以恢复原版。

2. **不要修改 `level2\config\` 目录下的任何文件**，`system_size.txt`、`system_fs_config`、`system_file_contexts`、`super_size.txt` 等是打包时的关键元数据，丢失或损坏后无法正确重建分区。

3. **不要修改或替换 `bootloader.PARTITION`**，烧录错误的 bootloader 会导致设备无法进入任何恢复模式（硬砖），需要专用工具短接 eMMC 芯片才能救回。

4. **新增文件必须同步更新权限配置**，`system_fs_config` 控制文件的 uid/gid/权限位，`system_file_contexts` 控制 SELinux 标签，缺少任一条目都可能导致应用无法运行或系统启动崩溃。

### 兼容性

5. **DDR.USB / UBOOT.USB 必须与硬件完全匹配**，不同芯片型号、不同内存颗粒规格的 u-boot 不可混用，否则烧录后内存初始化失败，设备变砖。

6. **Windows 下打包 ext4 存在局限**，`make_ext4fs` 的 Windows 版对 uid/gid 和 SELinux 标签的支持不完整，如遇打包后设备无法开机（但烧录成功无报错），优先考虑切换到 **WSL（Ubuntu）** 环境重新执行 Level 2 打包。

7. **super 分区有容量上限**，定制内容（尤其是预装应用）不能超过原始 super 分区的总容量，脚本会自动检测并报错，超出时需要删减内容。

### 工具限制

8. **dump.bat 和 dump_to_aml.bat 不支持 super 动态分区设备**（Android 10+ 动态分区），如需提取此类设备固件，需要使用其他专用工具或通过 TWRP 备份。

9. **pack_level1_fota.bat 不支持 super 动态分区**，仅适用于传统分区结构的固件（Android 9 及以下）。

10. **Rockchip 工具链相对独立**，`rkunpack.bat` / `rkpack.bat` 与 Amlogic 的工具流程相似但使用不同的底层工具（rkImageMaker、afptool），两者的 `level1\` 目录结构不同，不要混用。
